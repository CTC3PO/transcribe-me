A1

Custom Speech-to-Text PWA

Personal Wispr Flow alternative — mixed American-Vietnamese accent
3-day sprint Next.js Groq API MediaRecorder API ~$0/mo personal use

Objective

A web-based voice recorder that reliably transcribes a mixed American-Vietnamese accent cheaply. Hold-to-talk, instant copy to clipboard. Installable as a PWA on desktop and phone.

Why Groq + Whisper large-v3?

  • Groq free tier: 6,000 requests/day — more than enough for personal use
  • Whisper large-v3 natively handles Vietnamese/English code-switching
  • Latency ~0.5–1s vs OpenAI's 2–4s — feels instantaneous
  • Fallback: whisper.cpp locally if you hit rate limits

Day-by-Day Breakdown

1
ARCHITECT + BACKEND AGENT Scaffold & API Route

Create Next.js repo. Build /api/transcribe route using Groq SDK (groq.audio.transcriptions.create). Handle multipart audio upload, return JSON transcript. Store GROQ_API_KEY in .env.local. Test with curl before touching the frontend.

2
FRONTEND AGENT Recording UI & Browser API

Implement MediaRecorder Web API — hold button starts recording (sends in 3s chunks for speed), release stops. Build a large, touch-friendly hold-to-talk button in Tailwind. Implement real-time waveform visualization using the Web Audio API AnalyserNode (shows the mic is live). POST audio blob to /api/transcribe, display result immediately.

3
FRONTEND + QA AGENT Polish, PWA Config & History

Copy-to-clipboard button (with visual confirmation flash). Session history list using localStorage (last 20 transcriptions, no DB needed). Configure manifest.json + service worker for PWA installation. Test on mobile. Optionally: add language hint dropdown (English / Vietnamese / Auto) passed to Groq API.

Watch-out: Browser mic permissions behave differently on iOS Safari vs Chrome. Test on your actual phone early on Day 2 — don't leave mobile testing for Day 3.
Later (not MVP): Fine-tune Whisper on your own voice samples via HuggingFace. Add a Tauri wrapper for native desktop hotkey activation (like Wispr Flow's CMD+SHIFT+space). Keyboard shortcut for web version first.

Definition of Done

  • Speak 5 sentences mixing English + Vietnamese — all return within 2 seconds
  • Copy button works and shows visual confirmation
  • App is installable from Chrome on both Mac and Android
  • History persists across page refreshes via localStorage
B1

Transient City Item Exchange

Borrow/lend platform for expats and locals in Da Nang & Đà Lạt
4-day sprint Next.js Supabase Supabase Storage ~$0 on free tier

Objective

A marketplace where short-term expats and locals can list, borrow, and exchange items — drills, ladders, kitchen tools. No complex payments MVP. WhatsApp/Zalo contact flow to keep it simple and trusted.

DB Schema

  • profiles: id, display_name, phone, city, rep_score, created_at
  • items: id, title, description, image_url, category, owner_id, city, availability, price_type (free/borrow/sell)
  • borrow_requests: id, item_id, requester_id, status, created_at
  • (MVP: no in-app chat — WhatsApp link only)

Day-by-Day Breakdown

1
ARCHITECT + BACKEND AGENT Supabase Setup & Auth

Create Supabase project. Run SQL migrations for profiles and items tables. Enable Row Level Security (RLS) — only item owners can edit/delete their listings. Set up Google OAuth and/or Email magic link via Supabase Auth. Create Storage bucket item-images with public read access.

2
BACKEND AGENT Server Actions & Image Upload

Write Next.js Server Actions: createItem, getItems (with city filter), deleteItem. Build image upload utility: compress client-side with browser-image-compression before sending to Supabase Storage. Return public URL, store in items table.

3
FRONTEND AGENT Feed UI & Item Form

Build item card grid (photo, title, category badge, city, owner name). Filter bar: category + city + price type. Build "List an Item" form with image upload drag-and-drop. Basic profile page showing user's own listings.

4
FRONTEND + QA AGENT Contact Flow, Reputation & Polish

"Contact Owner" button generates a pre-filled WhatsApp/Zalo deep link (wa.me/{phone}?text=Hi, I'd like to borrow your {item_name}). Simple reputation: after a borrow, lender can +1 the borrower's score. Mobile layout pass. Add "Add to WhatsApp" share button for listings.

Watch-out: RLS policies are easy to misconfigure. Write a test script on Day 1 that verifies User A cannot delete User B's listing. Also: phone number storage — make sure users consent before storing, even for MVP.
Later: Stripe damage deposit hold on borrow requests. In-app chat (Supabase Realtime). Reputation tiers with borrowing perks. Cross-post listings to B4 (Da Nang events app) community board.

Definition of Done

  • User A signs up, uploads "Power Drill" with photo — card appears in feed
  • User B clicks Contact Owner — WhatsApp opens with pre-filled message
  • Filtering by "Da Nang" hides listings from other cities
  • Item images load in under 2 seconds on mobile
B4

Da Nang Event Aggregator

ĐàLạt.app ported for Da Nang — expats meet locals, WhatsApp chaos tamed
3-day sprint (fork of existing) Next.js Supabase Existing ĐàLạt.app codebase ~$0 on free tier

Objective

A centralized events feed for Da Nang — aggregating events that currently live buried in WhatsApp groups. Mobile-first. Public, no-login browse. Admin-only event input for MVP (open submission later).

The ĐàLạt.app Advantage

  • Architecture already proven — fork and rename
  • Supabase schema transfers almost identically
  • AI multilingual support already built
  • Main work: reskin + seed Da Nang-specific data
  • Estimated real build time: 1–2 days if codebase is clean

Day-by-Day Breakdown

1
ARCHITECT AGENT + HUMAN Fork, Reskin & Schema Migration

Fork ĐàLạt.app repo, update env vars, point to new Supabase project. Run migrations. Update categories for Da Nang context: Tech/Digital Nomad, Language Exchange, Sports/Running, Arts, Food. Add a source_group field to events table (which WhatsApp group it came from — useful for attribution later). Update branding, hero image, copy.

2
BACKEND + FRONTEND AGENT Admin Panel & Filtering

Secure admin-only event submission form (simple password or Supabase role). Ensure category filters work cleanly on mobile. Add "Submit Event" public form that goes into a pending queue (admin approves) — this is the key to community adoption. Seed with 10–15 real Da Nang events.

3
FRONTEND + QA AGENT Mobile Polish & Add-to-Calendar

"Add to Google Calendar" and "Add to Apple Calendar" (.ics file export) buttons on each event. Share button (Web Share API) so users can send events to their WhatsApp groups — this creates a viral loop. Final mobile layout pass. Deploy to Vercel.

Growth insight: The share button is the key feature. The whole product grows when people share events back into the same WhatsApp groups it aggregates from. Nail that UX on Day 3.
Later: WhatsApp bot integration (Twilio) to auto-ingest events. Mentorship matching feature (expat professionals ↔ local students). Language exchange pairing. Push notifications for saved categories.

Definition of Done

  • 5 seeded events visible on public page without login
  • Category filter works on mobile without page reload
  • Share button opens native mobile share sheet
  • "Add to Calendar" generates valid .ics file
  • Event submission form queues for admin approval
B2

Snapchat for Vietnamese Language

Buddy streak system + SRS flashcards — language learning as social accountability
4-day sprint Next.js Supabase Supabase Realtime Free tier sufficient for MVP

Objective

Two users pair as "buddies." Each completes a daily flashcard deck. They can see each other's streak. Miss a day — buddy gets notified. Start with a curated Vietnamese deck (100 most-used words). No teacher system in MVP.

Differentiation from Duolingo

  • Streak is tied to your buddy, not to a lonely owl's guilt trip
  • SRS algorithm (SM-2) vs Duolingo's gamified XP grind
  • No ads, no streaks-for-sale, no notifications designed to addict
  • Vietnamese-first content (Duolingo's Vietnamese course is shallow)

Day-by-Day Breakdown

1
ARCHITECT + BACKEND AGENT Schema & Auth

Tables: users, cards (front/back/audio_url/deck_id), decks, buddy_pairs (user_a, user_b, status), study_sessions (user_id, date, cards_reviewed, completed: bool). Supabase Auth (email). Seed 100 Vietnamese vocabulary cards with romanization.

2
BACKEND + FRONTEND AGENT Flashcard UI + SM-2 Algorithm

Core flashcard UI: show front, tap to flip, rate as Easy/Hard/Again. Implement SM-2 spaced repetition in a Server Action: Easy → next review in N*2.5 days, Hard → 1 day, Again → same day. Store next_review_date per card per user. "Today's deck" = all cards where next_review_date ≤ today.

3
FRONTEND AGENT Buddy Dashboard

Dashboard: your streak vs buddy's streak. Real-time buddy status ("Studying now" / "Done for today" / "Not yet") using Supabase Realtime subscriptions. Buddy invite system: generate a unique invite link, share via WhatsApp. Accept invite → buddy_pairs row created.

4
FRONTEND + QA AGENT PWA, Notifications & Polish

PWA manifest for home screen install. Email notification via Resend: if buddy finishes and you haven't by 8pm, send "Your buddy's waiting on you" email. Mobile swipe gesture for card flip (touch events). Full mobile layout pass.

Watch-out: Cold start / seeding is the biggest risk. Plan to personally onboard 5–10 buddy pairs from the Da Nang expat community before launch. The product only feels alive when your buddy is real.
Later: Audio pronunciation for each card (record native speaker, store in Supabase Storage). AI pronunciation scoring using Whisper (leverages A1 work). Teacher subscription system. Sentence-level cards beyond single words.

Definition of Done

  • User A and User B are linked as buddies via invite link
  • User A completes 10 flashcards — User B's dashboard shows "Done ✓"
  • Cards marked "Again" reappear in same session; "Easy" cards scheduled correctly
  • Buddy status updates in real-time without page refresh
C1

Pet Health CV Scanner

Upload a photo of your dog — AI flags potential conditions, prompts vet visit
3-day sprint Next.js Python / PyTorch HuggingFace Inference API Gradio (optional alt) HuggingFace free tier

Objective

Upload a photo of a dog's skin, eye, or ear → AI returns top-3 possible conditions with confidence scores → prominent "consult a vet" disclaimer. No human healthcare to avoid liability complexity.

Model Strategy

  • Fast path: Use HuggingFace Inference API with a pre-trained dermatology or animal image classifier (search dog skin disease on HF)
  • Better path: Fine-tune EfficientNet-B0 or ViT-base on the Dog Diseases Kaggle dataset
  • Fine-tuning in PyTorch ~2 hours on Colab free tier
  • Deploy fine-tuned model to HuggingFace Space as API

Day-by-Day Breakdown

1
ARCHITECT + BACKEND AGENT (Python) Model Fine-tuning & API

Download and explore a dog disease image dataset from Kaggle/HuggingFace. Fine-tune EfficientNet-B0 in PyTorch on Colab (use torchvision.models.efficientnet_b0(pretrained=True), replace classifier head). Train for 10 epochs (~2 hrs). Export to ONNX or push to HuggingFace Hub. Write a Next.js /api/scan route that forwards images to the HF Inference API and returns top-3 labels + confidence.

2
FRONTEND AGENT Drag-and-Drop Upload UI

Build upload zone using react-dropzone. Accept JPEG/PNG, max 10MB, client-side resize before upload. Animated skeleton loading state while API processes. Camera capture button for mobile (direct phone camera to upload).

3
FRONTEND + QA AGENT Results Display & Disclaimer

Results component: condition name, confidence bar, short plain-English description per condition (hardcoded for MVP, 10–15 conditions). Mandatory disclaimer modal on first visit (stored in localStorage). "Find a vet near me" button linking to Google Maps search. Mobile pass.

Watch-out: If you can't find a good pre-trained dog disease model on HuggingFace, fall back to GPT-4o Vision or Claude claude-sonnet-4-20250514's vision API for Day 1 — prompt it to describe potential conditions from the image. Less technically impressive but ships the product concept cleanly for validation.
Later: Multi-species (cats, birds). Symptom text layer alongside image. Vet clinic referral integration. Build a labeled dataset over time from user submissions (with consent) to improve the model.

Definition of Done

  • Upload a dog photo — API returns top-3 condition labels with confidence scores
  • Disclaimer modal appears on first visit and must be accepted
  • Mobile camera capture works on Android/iOS
  • Results render correctly even for very low confidence outputs (edge case)
C5

3D Reinforcement Learning Visualization

Live 3D playthrough of your Pacman Q-learning agent — portfolio centerpiece
3-day sprint Three.js / React Three Fiber Next.js Python (RL backend) $0 — all client-side

Objective

Take your existing Pacman Q-learning agent and build a real-time 3D visualization of its decision-making — Q-value heatmaps overlaid on the game board, agent movement, and reward signals rendered in Three.js. A rare, striking portfolio piece.

Two Paths (Choose One)

  • Path A (faster): Pre-record agent episodes as JSON replay files. Three.js plays them back with Q-value visualization — no live inference needed
  • Path B (impressive): WebSocket server (FastAPI) runs the RL agent live, streams state to Three.js frontend in real-time
  • Recommended: Path A for MVP (3 days), Path B as upgrade

Day-by-Day Breakdown

1
HUMAN + BACKEND (Python) Export Agent Replay Data

Modify your existing Pacman Q-learning agent to serialize game episodes as JSON: each step records {state, action, reward, q_values: {up, down, left, right}, agent_pos, ghost_positions, food_remaining}. Generate 5–10 complete episodes at different training stages (early, mid, converged). Save as episodes/ directory. This is the data layer for the visualization.

2
FRONTEND AGENT Three.js Scene Setup

React Three Fiber canvas in Next.js. Build the 3D Pacman grid: maze walls as BoxGeometry, floor tiles as PlaneGeometry. Load episode JSON and animate agent position using useFrame lerp. Color-code floor tiles as a heatmap based on Q-values (blue → red scale using d3-scale-chromatic). Add OrbitControls for free camera movement.

3
FRONTEND AGENT UI Overlays, Playback Controls & Polish

Floating HTML overlay (via @react-three/drei Html): current reward, episode number, training stage. Playback controls: play/pause, speed slider (0.5x–4x), step-by-step mode. Episode selector (Early Training vs Converged Agent) — dramatic diff in behavior is the "aha moment." Particle burst on reward events. Post-processing glow on agent using @react-three/postprocessing.

Portfolio framing: The real story is before vs after training. Let visitors toggle between an early-epoch agent (random, chaotic) and a converged agent (smooth, optimal). That contrast communicates what RL actually does better than any explanation.
Later / if Unreal assets exist: If you have 3D sets from your Gen AI for Virtual Environments class (classroom, island, etc.) in .glb format, substitute the Pacman grid with one of those environments. The RL visualization logic stays the same — just a more impressive backdrop.

Definition of Done

  • 3D scene loads in under 5 seconds in Chrome
  • Q-value heatmap visibly changes as agent moves
  • Early vs Converged episode comparison shows noticeably different behavior
  • Play/pause and speed controls work without breaking animation
  • Scene is navigable via mouse drag on desktop
C6

Urban Accessibility CV Prototype

Live camera + TensorFlow.js object detection → audio alerts for the visually impaired
4-day sprint (scoped down) React TensorFlow.js Web Speech API OpenStreetMap (optional) $0 — on-device inference

Realistic Scope for 4 Days

This is a proof-of-concept prototype, not a production accessibility tool. The 4-day goal is: live camera → detect obstacles → speak them aloud. Everything else (GIS integration, sensor data, full blind navigation) is future research scope. Be honest about this in the README.

Why It Still Matters

  • Strong ML portfolio project (real-time on-device inference)
  • Directly tied to your urban planning background
  • Proof-of-concept for a real research direction
  • TensorFlow.js COCO-SSD is well-documented and fast enough on mobile

Day-by-Day Breakdown

1
FRONTEND AGENT Camera Setup via WebRTC

React app with getUserMedia (rear camera default on mobile). Display live video feed. Handle iOS Safari's camera permission quirks early — this is always a surprise. Add a simple "Start" / "Stop" button.

2
FRONTEND AGENT TensorFlow.js Object Detection

Load @tensorflow-models/coco-ssd. Run inference on video frames every 500ms (balance accuracy vs battery). Draw bounding boxes on a canvas overlay. Filter to pedestrian-relevant classes: person, car, motorcycle, bicycle, bus, traffic light, stop sign. Display detection count per class.

3
FRONTEND AGENT Audio Feedback & Proximity Logic

Use Web Speech API (window.speechSynthesis) to speak detected objects. Throttle announcements: only speak if a class changes or confidence increases significantly (avoid speech spam). Add proximity estimation: larger bounding box = "close" → "Person very close ahead." Configurable voice speed and language.

4
BACKEND + QA AGENT (Optional) GPS Context Layer

Request GPS via Geolocation API. Query OpenStreetMap Overpass API for nearby crosswalks and bus stops. Announce them: "Crosswalk 20 meters ahead." This is genuinely hard to get right in 1 day — treat as stretch goal. Core value is in Days 1–3.

Honest scope warning: The original plan had this as a full navigation system. That's 6+ months of work. Frame this sprint as "a research prototype demonstrating real-time on-device computer vision for urban accessibility" — that framing is both accurate and compelling. The Day 4 GPS layer is aspirational.

Definition of Done

  • Live camera renders on mobile browser without lag
  • Person walking into frame triggers spoken "Person detected"
  • Bounding boxes drawn correctly in canvas overlay
  • No speech spam — announcements throttled appropriately
A2

Vibe-Coding Protocol (Personal Tool)

AI-powered project scaffold — plan → spec → execute → test, before writing a line of code
1-day setup (not a product) Python CLI Claude API Markdown templates Minimal API cost

Reframe

Don't build this as a product — use it as personal infrastructure. A Python CLI script you run at the start of every sprint. It takes a project brief and outputs: tech spec, Supabase schema SQL, file structure, day-by-day breakdown, and a test checklist. You've been writing these manually; automate them.

How to Use While Building Other Apps

  • Before starting A1: run python plan.py "speech-to-text PWA"
  • Output: auto-generated task.md for the sprint
  • Refine the output once, then use it as your agent instruction file
  • Iteration on the prompts = your real product development
Later (product path): Once you've used it for 5+ projects and refined the prompts, you'll have a genuinely opinionated workflow. Then it's worth building a proper UI and open-sourcing it. Your personal experience is the product's best validation.

Agent Workflow Protocol

How to use LLM agents effectively across every sprint above

Architect Agent

Designs the system. Creates task.md, DB schema, API contracts. Runs once at sprint start. Never writes UI code.

Backend Agent

Writes Server Actions, API routes, Supabase queries, Python scripts. Works from Architect's spec. Never touches JSX.

Frontend Agent

Builds React components, Tailwind layouts, browser APIs. Consumes Backend's endpoints. Owns all user-facing state.

QA Agent

Reviews code for bugs, edge cases, security issues (RLS, input validation). Runs at end of each day. Writes the Definition of Done checklist results.

Start each sprint: run the Architect Agent

Give it the project section from this document. Ask it to produce a task.md with the exact day breakdown, file structure, and Supabase schema SQL. This becomes your ground truth for the sprint.

You are the Architect Agent for a 3-day Next.js + Supabase sprint. Project: [paste project section from this document] Produce: task.md with day-by-day file list, Supabase schema SQL, and API contract (route names + input/output shapes). Be specific and opinionated. No hand-waving.
Each day: new agent context window

Start a fresh conversation for each agent. Paste the relevant day's task from task.md + any code already written. Never let one agent conversation grow beyond ~3 back-and-forths before starting fresh — context drift degrades output quality.

Backend before Frontend, always

Never let the Frontend Agent assume an API shape. Backend Agent ships the route first, you test it with curl or a simple fetch in the browser console, then Frontend Agent builds against the confirmed interface.

End each day: QA Agent review

Paste the day's new code to the QA Agent. Ask it to find: security issues (missing RLS, exposed env vars), edge cases (empty state, network failure), and anything that will break on mobile. Fix blockers before starting the next day.

You are the QA Agent. Review this Next.js + Supabase code for: 1. Security issues (RLS policies, exposed API keys, input validation) 2. Mobile browser edge cases 3. Empty/error states that aren't handled 4. Anything that will break in production Code: [paste]
Deploy at end of Day 1 (not Day 3)

Push to Vercel + connect Supabase env vars on Day 1, even if the app does almost nothing. You want to catch deployment-specific issues (env vars, build errors, CORS) early — not in the last hour of a sprint.
